并发的出现#

当处理器的速度大于I/O设备的速度的时候，并发就自然而然的出现了。并发也符合现实逻辑，现实世界无时无刻不在并发。

并发的问题#

共享内存是万恶之源。

  • 线程间共享
    • 完整的地址空间 (所有数据，包括代码)
    • 所有指令取指、执行都作用在同一个地址空间
  • 线程独立拥有
    • 寄存器
    • 堆栈

问题1：原子性#

CPU指令的非原子性： load 和 store

int pay(int t) {
if (balance >= t) {
balance -= t;
return SUCCESS;
} else {
return FAIL;
}
}
#define n 100000000
long sum = 0;
void do_sum() { for (int i = 0; i < n; i++) sum++; }
void print() { printf("sum = %ld\n", sum); }
int main() {
create(do_sum);
create(do_sum);
join(print);
}

并发会导致原子性的丧失，load 和 store 不是原子操作时，容易得到错误的计算结果。

问题2：顺序性#

编译器优化。

分两个线程，计算 1+1+1+…+1 (共计 2n 个 1)

#define n 100000000
long sum = 0;
void do_sum() { for (int i = 0; i < n; i++) sum++; }
void print() { printf("sum = %ld\n", sum); }
int main() {
create(do_sum);
create(do_sum);
join(print);
}

开启编译优化

  • -O1: 100000000
  • -O2: 200000000

编译器对程序的改写 (涉及内存访问指令的移动或删除) 仅对单线程执行有正确性保证。

问题3：可见性#

为了使 CPU 运行得更快，CPU 可以不按顺序执行指令！

movl $1, (x) # x = 1, cache miss
# 如果等这条指令执行完，会浪费大量时间
movl (y), %eax # CPU 会立即执行这条指令
# (此时 y = 0)
...

现代处理器 (动态数据流分析)：

  • 如果两条指令没有数据依赖关系，就让它们并行执行好了！
  • 乱序执行 (out-of-order)
    • 多处理器上执行的结果可以不等价于指令按照某个顺序执行的结果
    • Further reading
int x = 0, y = 0;
void thread_1() {
x = 1; printf("y = %d\n", y);
}
void thread_2() {
y = 1; printf("x = %d\n", x);
}

实验结果 (4 x Xeon X7460, 24-cores)

打印的 x打印的 y概率
000.2% (难以接受这个设定啊)
0182.3%
1017.5%
110.0%

生产者-消费者模型#

生产者 (线程) 生产资源 (一个对象)，生产时间不确定

  • 缓冲区里没有空余的空间存放生产的对象时等待

消费者 (线程) 消费资源 (取走一个对象)，消费时间也不确定

  • 缓冲区里没有对象可以消费时等待
